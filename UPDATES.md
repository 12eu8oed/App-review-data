# 🚀 앱 리뷰 수집기 업데이트 기록 (Update Log)

## 📅 2026-05-03: App Store 리뷰 수집 한계 극복 업데이트

### 1. [오류] 애플 App Store RSS API 한계 및 불안정성
- **현상**: 애플에서 제공하는 기존 RSS 피드(`itunes.apple.com/kr/rss/...`)가 최신 리뷰를 누락하거나, 최대 500개까지만 가져올 수 있는 물리적 한계가 존재함. 또한 최근 애플의 API 변경으로 인해 데이터가 빈 값으로 오는 경우가 잦아짐.
- **영향**: 사용자 요청 시 500개 이상의 리뷰를 조회할 수 없었고, 종종 "리뷰가 없습니다"라는 오류 메시지가 출력됨.

### 2. [해결] Web Scraping(HTML) 우회 및 병렬 수집 구현
- **해결책**: RSS API 대신 애플 앱스토어 웹 페이지의 실제 HTML을 파싱하는 방식으로 로직 전환.
- **고도화**: 
    - `cheerio` 라이브러리를 도입하여 서버 사이드에서 HTML 구조 분석.
    - 페이지네이션을 수동으로 탐색하여 500개 이상의 과거 데이터까지 접근 가능하도록 설계.
    - `Promise.all` 기반의 Batch Fetching(8개 페이지 동시 수집)을 통해 수집 속도 대폭 개선.

### 3. [오류] 데이터 중복 및 개수 불일치 이슈
- **현상**: 웹 페이지 스크래핑 시 동일한 리뷰 데이터가 HTML 내 여러 노드에 중복 배치되어 있어, 1000개를 요청해도 중복을 제거하면 500개만 남는 현상 발생.
- **해결책**: 
    - 글로벌 중복 제거(Deduplication) 필터를 강화하여 작성자 이름과 본문이 완벽히 일치하는 데이터는 필터링.
    - 중복이 발생할 것을 대비해 실제 요청 개수보다 20% 더 많은 페이지를 미리 탐색하도록 로직 수정.

### 4. [기능 추가] 수집 가능 개수 상향 (Max 5,000)
- **업데이트**: 기존 50~500개였던 프리셋 범위를 **100, 500, 1000, 2000, 5000개**로 대폭 상향.
- **UI 개선**: 사용자가 앱스토어에서도 대량 수집이 가능함을 알 수 있도록 안내 문구 및 아이콘 업데이트.

## 📅 2026-05-05: App Store 대량 수집 시 누락 현상 수정

### 1. [오류] 5000개 요청 시 2600여 개만 수집되는 이슈
- **현상**: 지그재그(또는 다른 앱)의 애플 앱스토어 리뷰를 5,000건 요청하면 약 2,600건 부근에서 멈추는 현상 발견. 구글 플레이스토어는 5,000건 정상 조회됨.
- **원인 분석**: 
    - 애플 서버(Akamai CDN)가 동일 IP에서 단시간에 반복적으로 페이지네이션을 요청할 경우 랜덤하게 **403 Forbidden (접근 거부)** 에러를 반환.
    - 특히 250페이지(약 2500개 분량)를 넘어가는 시점부터 연속적으로 403 에러가 반환되며 빈 데이터를 내려주어, 시스템이 "더 이상 리뷰가 없다"고 판단하고 조기 종료(Break)된 것임.

### 2. [해결] 페이지 최대 한도 상향 및 403 연속 에러 내성 추가
- **해결책**:
    - 기존에는 페이지당 무조건 8~10개가 있다고 가정하여 `Limit(요청개수/8)` 페이지까지만 탐색하도록 보수적으로 제한했으나, **최대 1,000페이지**까지 여유 있게 탐색 범위를 확충 (`Math.min(1000, Math.ceil(num / 5))` 적용).
    - 수집 도중 일시적인 네트워크 차단(403)으로 인해 `빈 데이터(0건)`가 내려오더라도 즉각 종료되지 않도록 안전장치 구비.
    - 빈 데이터가 5번 연속(`emptyBatchCount >= 5`) 수신될 때만 최종적으로 더이상 응답이 없다고 판단하도록 **Grace Period 로직** 추가 적용.

### 3. [한계 안내] 애플 고유의 최대 출력 한계
- 애플 앱스토어의 자체적인 HTML 페이지네이션 구조 한계 상, 통상 **300페이지 ~ 400페이지(약 3,000건~4,000건 내외)** 이상의 과거 리뷰는 직접적인 물리 접근을 완전히 닫아버리는 정책이 있습니다. 
- 방어 로직 추가: 너무 빠른 속도로 한 번에 페이지네이션을 돌면 애플 서버가 IP를 차단하게 되므로, Batch 당 처리 개수를 10개에서 5개로 줄이고 각 배치 사이에 100~300ms 의 랜덤한 딜레이를 주어 사람이 수동 탐색하는 것처럼 보이게 우회.
- RSS 페일오버(Fallback): 만약 단기간 IP 차단으로 인해 Web Scraping에 100% 실패하더라도, 에러를 곧바로 던지지 않고 기존 RSS API를 통해 최소 500개는 가져올 수 있도록 Fallback(플랜B) 함수 실행을 보장.
